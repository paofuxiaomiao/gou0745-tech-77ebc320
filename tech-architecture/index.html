0745购项目合作实施方案当前以 React + Vite + Tailwind CSS 构建为全屏翻页式 Web Deck， 发布到 GitHub Pages。它的核心不是“单页堆内容”，而是把演示拆成三层：页面运行时、共享视觉组件、具体 slide 内容。

为了让项目可维护可升级，后续编辑应遵守一个简单原则：内容改 slide，框架改 Deck，视觉规范改 SlideKit 与 index.css，发布只动 gh-pages 产物。 这样每次更新不会把业务文案、动画系统、资源路径和部署配置搅在一起。

当前公开页面：https://paofuxiaomiao.github.io/gou0745-tech-77ebc320/。公开资源路径使用 /gou0745-tech-77ebc320/manus-storage/，一旦发布，文件名应视为稳定接口。

Project Advantage · Expression Manufacturing

项目优势：把 AI 退到幕后，把企业表达推到极致

我们追求的不是“快速生成一套看起来像 PPT 的图片”，而是把企业展示中的每一次表达做到极致： 从字体选择、字重关系到网格、留白与信息节奏，每一个细节都经过可复用、可校准的系统设计。

01 字体即结构

字体不是最后套上的皮肤。字族、字重、字号、行距与中英文比例共同建立信息层级，让企业语言稳定、克制且可信。

02 排版即叙事

页面通过网格、留白、对齐与节奏组织观点，不依赖一张大图掩盖信息问题，让每一页都能承担明确的沟通任务。

03 批量而不复制

用结构化内容、设计 token、共享组件与页面注册表形成生产线，既能批量生成企业展示 PPT，又能保持逐页差异和统一品质。

04 不同于图片生成

页面由真实文字、布局规则和浏览器渲染构成，而非不可编辑的文生图海报；内容可维护、版式可升级、资产可追溯。

这条路径更像冶炼金属：结构化内容是矿石，组件与规则是模具，渲染、校准与复核是反复锻造。 最终留下的是稳定、精确、可持续生产的企业表达，AI 痕迹则在这条生产链中被逐层消除。

因此，它既不同于传统 PPT 的逐页手工拼装，也不同于依赖图片生成的海报式生产。 我们用代码定义版式系统，用组件沉淀设计语言，用数据驱动页面批量生产，再由人工审美完成最后校准。

技术栈不是随便选的，而是围绕“演示型 Web Deck”服务

这套技术栈的目标是：浏览器内翻页顺滑、视觉语言可复用、图片资产可长期归档、静态发布成本低。它不依赖后端运行时即可公开访问。

页面架构：一条主线，四个职责边界

页面主体由 Home.tsx 组织 slide 顺序，由 Deck.tsx 处理运行时导航，由 SlideKit.tsx 承担共享视觉骨架，具体内容分散在 SlidesA-J.tsx。

后续维护时，不建议在 Deck.tsx 写业务页面逻辑，也不建议在 slide 文件里重复实现页眉、页码、章节标题。 这类重复逻辑应该收敛到 SlideKit.tsx 或 index.css。

资产与发布：公开页最怕的不是代码错，而是资源路径错

源仓库存档在 assets/，运行时页面引用的是 /manus-storage/<文件名>。 GitHub Pages 是项目子路径，所以构建后必须把资源路径重写为 /gou0745-tech-77ebc320/manus-storage/。

  1. 源素材：存放在 assets/images/generated、retouched、extracted_from_ppt。
  2. 运行引用：源码中使用 /manus-storage/xxx_hash.png。
  3. 静态产物：构建后复制到 dist/public/manus-storage/。
  4. 公开访问：GitHub Pages 上变为 /gou0745-tech-77ebc320/manus-storage/xxx_hash.png。

升级策略：允许演进，但每层只动自己的事

这个项目后续要升级，最重要的是分层升级。文案、视觉、依赖、发布链路不要混在一笔提交里。每个变更都要能回答： “我改的是内容、框架、视觉系统、资产，还是部署？”

推荐升级顺序

  1. 内容更新：只改 Slides*.tsx，必要时改 Home.tsx slide 顺序。
  2. 视觉更新：优先改 index.css tokens，再改 SlideKit.tsx。
  3. 框架更新：单独升级 Vite、Tailwind、React 或 wouter，逐项验证。
  4. 发布更新：只动 gh-pages 产物，验证首页和关键图片 URL。

可维护性的最终判断：新人能不能接手

这个页面未来最常见的维护动作会是：替换某一页文案、增加一页对比稿、修一张图、重新发布。架构说明的价值，就是让接手者知道该去哪个文件改，改完跑什么命令，发布后检查哪个 URL。

  • 新增页面：新增 slide 组件，注册到 Home.tsx。
  • 改全局动效：修改 Deck.tsx 与 index.css。
  • 改设计语言：修改 token、字体、通用组件，不在单页里散落写死。
  • 改素材：原始素材归档，发布文件原名替换，线上 HEAD 检查体积。
  • 改部署：先在临时目录生成静态产物，再推 gh-pages。
